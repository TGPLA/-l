# 项目规范守护者

本 Skill 确保所有项目操作严格遵守既定规范，防止违规操作。

## 核心规则

### 1. 操作前必查

在执行任何文件修改或删除操作前，**必须**先完成以下步骤：

1. **查阅项目地图**：阅读根目录下的 `项目地图.md`，了解文件的功能定位和归属
2. **查阅规则文件**：阅读 `.trae/rules/` 目录下的所有规则文件
3. **确认文件状态**：检查目标文件是否已标记 `// @审计已完成`

### 2. 归档流程

当需要删除或废弃文件时，**必须**执行归档流程：

1. **物理删除**：仅删除确认无引用的孤儿文件
2. **逻辑归档**：将有历史价值的文件移动到 `archived_旧版归档/` 目录
3. **地图同步**：在 `项目地图.md` 的【归档区】记录归档信息
4. **审计同步**：如有违规清单，同步更新 `审计违规清单.md`

### 3. 新建文件规范

创建新文件时，**必须**包含：

```typescript
// @审计已完成
/**
 * 文件名：xxx.ts
 * 中文名：xxx
 * 功能描述：xxx
 */
```

### 4. 行数限制

- 所有文件**必须**控制在 100 行以内
- 超过 100 行的文件**必须**拆分为多个模块
- 拆分遵循三层架构：数据逻辑层 + 子组件层 + 主视图层

### 5. 中文命名规范

- 文件名：优先使用中文拼音（如 `ShuJiTiShiCiPeiZhi.ts`）
- 函数名：使用中文（如 `huoQuPeiZhi`）
- 变量名：使用中文（如 `shuMing`）
- 注释：使用中文

## 目录结构规范

```
src/
├── api/                    # API 层
│   └── business/           # 业务 API 模块
├── components/             # 组件层
│   └── business/           # 业务组件模块
├── config/                 # 配置层
│   └── business/           # 业务配置模块
├── hooks/                  # Hooks 层
│   └── business/           # 业务 Hooks 模块
├── services/               # 服务层
│   └── business/           # 业务服务模块
├── store/                  # 状态存储层
│   └── business/           # 业务存储模块
├── types/                  # 类型定义层
│   └── business/           # 业务类型模块
└── utils/                  # 工具层
    └── business/           # 业务工具模块
```

## 违规处理

如果发现违规操作：

1. **立即停止**：暂停当前操作
2. **报告违规**：向用户说明违规内容
3. **提供方案**：给出合规的替代方案
4. **执行修正**：在用户确认后执行合规操作

## 检查清单

操作前确认：

- [ ] 已阅读 `项目地图.md`
- [ ] 已阅读 `.trae/rules/` 规则
- [ ] 目标文件行数 < 100
- [ ] 新文件包含中文头部注释
- [ ] 新文件包含 `// @审计已完成` 标记
- [ ] 归档操作已同步到地图和审计清单
