# 功能完整性校验规则 (Functional Integrity Rules)

## 1. 核心哲学

禁止仅凭"代码逻辑已写完"或"文档标记已完成"就判定任务结束。必须从用户视角出发，验证**"状态 → 业务逻辑 → UI 控制 → 用户交互"**的完整链路。

---

## 2. 功能上线五要素 (Mandatory Checklist)

在提交任何新功能或标记功能为"已完成"之前，必须逐一核对以下五个环节：

| 要素 | 说明 | 检查方式 |
|------|------|---------|
| **状态定义** | 状态（如 youJianCaiDan）是否已在合适的 Hook 或 Store 中定义？ | 搜索状态变量名 |
| **业务逻辑** | 该状态是否已正确驱动业务行为（如 onContextMenu 的拦截）？ | 检查状态使用处 |
| **UI 控制入口** | 用户是否能在界面上看到控制该功能的按钮、开关或菜单？ | 浏览器中查看界面 |
| **交互闭环** | 用户操作 UI 后，状态是否实时改变？业务行为是否随之改变？ | 实际点击测试 |
| **持久化** | 用户刷新页面或重启后，该设置是否依然生效（如 localStorage）？ | 刷新后验证 |

---

## 3. "端到端"检查路径

在进行功能自测时，严禁使用"检查逻辑代码"代替"检查用户路径"。

| 错误路径（逻辑检查） | 正确路径（端到端检查） |
|---|---|
| useHook.ts 有状态 → ✅ | 状态 → 传递至组件 → 渲染 UI → 用户点击 → ✅ |
| 业务逻辑包含 if(enabled) → ✅ | 禁用开关 → 触发业务行为 → 验证是否拦截 → ✅ |
| 参考项目说明标记为"完成" → ✅ | 独立启动环境 → 模拟用户操作 → 验证结果 → ✅ |

---

## 4. 组件引用与透传规范

为避免"UI 遗漏"再次发生，遵循以下组件开发规范：

### 主动追踪数据流

在创建控制逻辑 Hook 时，必须立即搜索对应的 View 层组件（如 GongJuLan、KongZhiQu），并完成 Props 的透传。

### 占位符防御

如果 UI 暂时未实现，必须在组件代码中留下 `// TODO: UI control for [feature name]` 注释，并在开发文档中明确标记"UI 缺失"。

### 引用验证原则

新功能涉及的底层组件修改后，必须向上回溯至页面级组件（如 EPUBReader.tsx），确保引用链路未断裂。

---

## 5. AI 协同提示 (For Trae/Copilot)

当要求 AI 检查功能完整性时，请使用以下指令：

> "请根据功能完整性校验规则，检查 [功能名称] 的端到端链路。不仅要看逻辑层实现，还要核实 UI 控制入口是否已接入，并确认 Props 是否从状态源透传到了最末端的交互按钮。"

---

## 6. 失败案例

**EPUB 右键菜单功能遗漏**：

| 要素 | 状态 | 问题 |
|------|------|------|
| 状态定义 | ✅ | `useYueDuMoShiSheZhi.ts` 已定义 |
| 业务逻辑 | ✅ | `useEPUBReaderShiJian.ts` 已实现 |
| UI 控制入口 | ❌ | 工具栏无开关按钮 |
| 交互闭环 | ❌ | 用户无法操作 |
| 持久化 | ✅ | localStorage 已配置 |

**根因**：只检查了前两项，跳过了 UI 入口和交互验证。

**教训**：代码存在 ≠ 功能可用，必须端到端验证。
