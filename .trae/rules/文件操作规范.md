# 文件操作规范

## 核心原则

**行数不是质量指标，复杂度才是。**

---

## 修改文件前

1. **关键代码检查**（修改核心逻辑时）
   - 搜索 `@关键代码` 标记，查看 `.trae/关键代码清单.md`
   - 理解原因后再修改，修改后验证功能
2. 查阅 `02_项目地图.md` 确认文件定位
3. 检查是否有 `// @审计已完成` 标记

## 删除文件前

1. **关键代码检查**（必须）
   - 搜索 `@关键代码` 标记，查看清单，理解原因
2. 全局搜索确认无引用
3. 物理删除，同步更新 `02_项目地图.md`【归档区】

## 创建文件时

1. 中文头部注释 + `// @审计已完成` 标记
2. **无论来源**（复制/参考/AI生成）都必须遵守规则
3. **新旧代码**：新代码产生时询问是否删除旧代码

---

## 文件大小限制（按类型）

| 文件类型 | 推荐限制 | 硬性红线 | 说明 |
|---------|--------|---------|------|
| **页面组件**（-Page.tsx） | ≤300 行 | 无上限 | 路由入口，需要完整状态 |
| **功能组件**（-Modal.tsx） | ≤200 行 | 无上限 | 弹窗组件，逻辑较完整 |
| **业务 Hooks**（useXxx.ts） | ≤250 行 | 无上限 | 状态管理，可较长 |
| **UI 组件**（Button.tsx） | ≤100 行 | ≤150 行 | 基础组件，必须精简 |
| **工具函数**（xxx.ts） | ≤150 行 | ≤200 行 | 纯函数库 |
| **类型定义**（types.ts） | ≤80 行 | ≤100 行 | 类型声明 |

---

## 复杂度限制（真正重要的指标）

```
🚨 硬性红线（必须遵守）：

1. 单函数 ≥ 30 行          → 必须拆分
2. 嵌套层级 ≥ 4 层        → 必须重构  
3. 同行 import ≥ 12 个    → 必须拆分模块
4. 条件分支 ≥ 8 个        → 必须拆分函数
```

```
✅ 推荐标准（鼓励遵守）：

1. 单一职责：每个文件只做一件事
2. 内聚性：相关逻辑放一起，不相关拆分
3. 接口清晰：导出明确，依赖明确
```

---

## 检查方式

```bash
npm run check:lines        # 显示所有文件行数（参考用）
npm run build             # 构建验证
# 复杂度需要人工审查，非自动化
```
