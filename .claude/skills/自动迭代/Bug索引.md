# Bug索引 - 智能复用核心数据库
> **用途**：当遇到Bug时，先查此索引匹配多个历史方案，根据场景选择最合适的解决方案，避免重复踩坑。
> **更新时机**：每次复盘存档创建/更新时自动同步。
> **核心特性**：支持多次修复失败记录，每个Bug可追踪修复历史

## 核心特性：AI自动尝试 + 多方案并存 + 修复历史追踪

```json
{
  "Bug ID": {
    "问题名称": "...",
    "AI尝试历史": [
      { "次数": 1, "方案": "方案A", "结果": "失败", "原因": "...", "时间": "...", "执行者": "AI" },
      { "次数": 2, "方案": "方案B", "结果": "失败", "原因": "...", "时间": "...", "执行者": "AI" },
      { "次数": 3, "方案": "方案C", "结果": "成功", "原因": "...", "时间": "...", "执行者": "AI" }

    "解决方案": [

      { "序号": "方案B", "方法": "...", "适用场景": "...", "优先级": "...", "状态": "...", "AI尝试历史": ["AI尝试2次后成功"] }
    ]
  }
}
```

**核心原则**：
- ✅ **AI自动尝试修复方案**，用户只需说"不行"或"没解决"
- ✅ **每个Bug可以有多个解决方案并存**
- ✅ **记录每次AI修复尝试的结果**（成功/失败/部分成功）
- ✅ **追踪AI尝试历史**，避免AI重复尝试已失败的方案
- ✅ **连续3次AI尝试失败触发深度复盘**（调用失败复盘存档技能）
- ✅ **每个方案标注适用场景**，帮助AI选择
- ✅ **标注优先级**：primary（首选）> alternative（备选）> fallback（最后手段）
- ✅ **标注状态**：在用（当前使用）| 废弃（历史方案）| 实验（测试中）
- ✅ **保留废弃方案的历史记录**，作为参考或回退方案

### AI自动尝试流程

```
用户遇到Bug → 用户描述问题
    ↓
AI查Bug索引 → 匹配历史方案
    ↓
AI自动尝试方案A（primary）
    ↓
用户验证 → 说"不行"或"没解决"
    ↓
AI自动记录失败 → 自动尝试方案B（alternative）
    ↓
用户验证 → 说"还是不行"
    ↓
AI自动记录失败 → 自动尝试方案C（fallback）或新思路
    ↓
用户验证 → 说"还是不行"
    ↓
🚨 AI自动触发【失败复盘存档】技能
    ↓
生成深度复盘报告 + 求助式接力语
```

### 用户只需做的3件事

1. **描述问题**："我遇到了..."
2. **验证结果**："不行" / "没解决" / "还是不行"
3. **确认成功**："解决了" / "可以了" / "修复了"

其他所有事情（查索引、选方案、写代码、记录历史）都由AI自动完成！

## 修复中出现新Bug处理流程（重要！）

### 场景描述
在修复Bug A的过程中，Bug A消失了，但出现了新的Bug B。这种情况很常见，通常是因为：
- 修复方案引入了新的问题（副作用）
- 修复方案暴露了之前隐藏的问题
- 修复方案改变了代码逻辑，触发了其他bug

### AI自动执行流程

```
用户反馈："之前的Bug没了，但出现了新问题"
    ↓
【AI自动执行 - 修复中出现新Bug处理】
├── 1. 确认原Bug A状态
│   ├── 标记Bug A为"✅ 已解决"（如果确实消失）
│   └── 记录解决方式："修复过程中自然消失"
├── 2. 识别新Bug B
│   ├── 提取Bug B的关键词和症状
│   ├── 判断Bug B是否与Bug A相关
│   └── 判断Bug B是否是修复方案的副作用
├── 3. 查Bug索引
│   ├── 先查Bug B是否有历史方案
│   ├── 如果有 → 按正常流程处理
│   └── 如果没有 → 标记为新Bug
├── 4. 记录关联关系
│   ├── 在Bug A记录中标记："修复后出现关联Bug B"
│   └── 在Bug B记录中标记："由修复Bug A引发"
├── 5. 处理Bug B
│   ├── 如果是副作用 → 尝试调整修复方案
│   ├── 如果是新问题 → 正常查索引、尝试方案
│   └── 如果是隐藏问题 → 分析触发条件
└── 6. 生成处理报告
    ├── 原Bug A状态
    ├── 新Bug B分析
    └── 建议处理策略
```

### 不同情况的处理策略

| 情况 | 判断标准 | 处理策略 |
|------|---------|---------|
| **副作用** | 新Bug与修复方案直接相关 | 调整修复方案，消除副作用 |
| **暴露隐藏问题** | 新Bug之前被掩盖，现在显现 | 独立处理新Bug，正常查索引 |
| **连锁反应** | 修复A导致B，修复B导致C | 记录连锁关系，寻找根本解决方案 |
| **误判** | 原Bug其实没消失，只是表现变了 | 重新分析，合并为同一个Bug |

## 修复成功处理流程（重要！）

### 触发条件
用户说以下任一表达时触发：
- "修复了" / "解决了" / "可以了" / "没问题了"
- "好了" / "搞定了" / "成功了"
- "验证通过" / "测试通过"

### AI自动执行流程

```
用户说"修复了"
    ↓
【AI自动执行 - 修复成功处理】
├── 1. 识别当前修复的Bug（从对话上下文）
├── 2. 记录修复成功到 Bug索引.AI尝试历史
│   ├── 标记最后尝试的方案为"成功"
│   ├── 记录成功时间
│   └── 记录成功方案的详细说明
├── 3. 更新方案状态
│   ├── 成功方案 → 状态改为"🟢 在用"
│   ├── 成功方案 → 优先级改为"⭐ primary"
│   └── 其他方案 → 状态改为"🔴 废弃"或保留为备选
├── 4. 更新修复历史
│   ├── 添加成功记录到"AI尝试历史"
│   └── 更新"解决状态"为"✅ 已解决"
├── 5. 【可选】更新技能文档
│   ├── 询问用户是否同步到技能速查表
│   └── 如果同意，更新对应技能的解决方案

    ├── 方案详情（核心代码、适用场景）
```

```typescript
// 核心思路：使用 epubjs 原生的 rendition.next() 翻页，而不是 spine 跳转

const handleNextPage = useCallback(() => {
  const rendition = renditionRef.current
  if (!rendition) {
    return
  }

  // 使用 rendition.next() 翻到下一页（处理章节内容跨页的情况）
  rendition.next().then(() => {
    const currentLocation = rendition.location
    const currentHref = currentLocation?.start?.href || ''
    if (currentHref && saveImmediately) {
      saveImmediately(currentHref)
    }

    if (currentHref && saveImmediately) {
```

## 匹配算法说明

### 当遇到Bug时的匹配流程

```
1. 用户描述问题
   ↓
2. 提取问题关键词
   - 例如："划线在不同浏览器看不到"
   - 提取关键词：划线, 同步, 浏览器, 不同步
   ↓
3. 匹配 Bug 索引
   - 关键词 "划线" + "同步" + "浏览器" → 匹配到 B001
   ↓
4. 检查修复历史
   - B001: 已尝试3次，第3次成功
   - 展示："此问题曾有3次修复记录，最终方案为..."
   ↓
5. 提供历史方案（按优先级排序）
   ├─ ⭐ primary → 推荐使用（已验证成功）
   ├─ ⚠️ alternative → 备选方案
   └─ 🔴 fallback → 最后手段
   ↓
6. 用户选择方案并尝试修复
   ↓
7. 修复结果反馈
   ├─ ✅ 成功 → 更新修复历史，标记方案状态
   ├─ ❌ 失败 → 记录失败原因，尝试备选方案
   └─ 🔄 部分成功 → 记录情况，继续优化
   ↓
8. 如果连续3次失败
   → 触发【失败复盘存档】技能
   → 生成深度复盘报告
   → 生成求助式接力语
```

### AI自动尝试失败的处理策略

| AI尝试次数 | 策略 | AI自动执行 | 用户需要做的 |
|-----------|------|-----------|-------------|
| 第1次失败 | 记录 + 自动尝试备选 | 更新AI尝试历史，自动尝试 alternative 方案 | 说"不行" |
| 第2次失败 | 记录 + 自动尝试最后手段 | 更新AI尝试历史，自动尝试 fallback 方案或新思路 | 说"还是不行" |
| 第3次失败 | 🚨 自动触发复盘 | 自动调用失败复盘存档技能，生成求助式接力语 | 说"还是不行" |
| 第3次以上 | 暂停修复 | 建议寻求外部帮助或暂时绕过 | 确认是否继续 |

## 自动同步配置

此索引由自动迭代技能维护：

- **触发时机**：
  - 复盘存档目录出现新文件时
  - Bug修复成功/失败时
  - 用户要求更新时
- **更新方式**：增量更新索引表 + 修复历史
- **保留历史**：旧索引移入 `索引历史/` 目录

### 配置参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 索引路径 | `.trae/skills/自动迭代/Bug索引.md` | 索引文件位置 |
| 存档目录 | `archived_复盘存档/` | 复盘存档源目录 |
| 历史目录 | `.trae/skills/自动迭代/索引历史/` | 旧索引归档 |
| 最大修复次数 | 3 | 超过则触发复盘 |
| 复盘技能 | `失败复盘存档` | 连续3次失败时调用 |
